# 한국어 보존적 퇴고 플레이북

## 판단 순서

1. 문장이 틀렸는지가 아니라 독자에게 실제로 불편한지 확인한다.
2. 규칙에 맞아도 장르와 문맥에 필요한 표현이면 유지한다.
3. 삭제, 짧은 치환, 문장 재구성 순으로 작은 수정을 먼저 시도한다.
4. 수정 전후의 사실, 주체, 시제, 부정, 조건과 인과 관계를 대조한다.

## 문서별 기준

### 업무 문서

- 결론과 요청 사항을 앞에 두되 원문에 없는 의사결정이나 기한을 만들지 않는다.
- PR, Linear, Slack의 제목·체크박스·식별자·링크를 보존한다.
- 사용자가 문체 편집을 요청하지 않았다면 해당 도구의 전용 스킬에 맡긴다.

### 기술 문서

- 코드, 명령, API 이름, 환경변수, 경로, 오류 메시지를 그대로 보존한다.
- 반복처럼 보이는 용어라도 개념 구분에 필요하면 줄이지 않는다.
- 모호한 대명사는 실제 대상을 확인할 수 있을 때만 명시한다.

### 보고서·제안서

- 추측을 사실로 강화하거나 확정 표현을 완곡하게 약화하지 않는다.
- 근거 없는 '중요하다', '혁신적이다'를 삭제할 때 문장의 주장 자체가 사라지지 않는지 확인한다.
- 열거 구조가 비교 기준이나 요구사항을 나타내면 유지한다.

### 블로그·에세이

- 문장 길이를 기계적으로 다양화하지 않는다.
- 개인적인 말투, 반복, 파편 문장이 의도된 리듬이면 보존한다.
- 브랜드 문구와 CTA는 사용자가 변경을 요청하지 않는 한 유지한다.

## 규칙별 주의점

- `~를 통해`: 수단이 핵심이면 정상 표현이다. 모든 사례를 `~로`로 바꾸지 않는다.
- `것이다`: 설명·강조·회고의 기능이 있으므로 빈도만으로 제거하지 않는다.
- 피동: 행위자가 중요하지 않거나 알려지지 않은 경우 자연스럽다.
- 접속어: 논리 관계를 독자가 오해할 수 있다면 삭제하지 않는다.
- 불릿과 헤딩: 구조화된 업무 문서에서는 산문보다 적합할 수 있다.
- 전문용어와 영어: 번역했을 때 의미가 달라지거나 검색성이 떨어지면 유지한다.

## 검증 한계

결정적 스크립트는 문자열 수준의 소실과 형식 변화를 찾는다. 다음 항목은 원문과 직접 대조한다.

- 새로운 주장이나 근거 추가
- 행위자와 대상의 교체
- 부정, 조건, 예외의 소실
- 시제와 확실성 수준 변화
- 같은 수치를 다른 대상에 연결한 경우
- 고유명사의 비슷한 철자 변경

## 출력 방식

수정본을 먼저 보여준다. 변경 설명은 사용자가 검토에 도움을 받을 정도로만 제공하고, 규칙 ID나 진단 수치를 기본 출력으로 노출하지 않는다.
